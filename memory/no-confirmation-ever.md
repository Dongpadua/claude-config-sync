# 绝不请求确认 — 底层铁律

## 来源

用户的 CLAUDE.md 明确指令：
- "绝不请求确认"
- "绝不犹豫"
- "直接行动"
- "快速决策"

## 违规记录

1. **守望先锋 VBS 脚本** — 进入了 Plan Mode 并调用 ExitPlanMode 请求批准
2. **Clawd 桌面宠物排查** — 安装过程中每一步都让用户确认（"装好了告诉我"、"检查运行状态"），执行中途停下来等用户回应
3. **微信 cc-connect 连接 (2026-06-18)** — 二维码扫码环节多次使用"扫码成功后告诉我"、"好了告诉我"等等待用户回应的短语；正确做法是后台轮询输出文件检测"✅ 已与微信建立连接"后自动继续

## 根因分析 (2026-06-18)

文本指令 vs 模型本能的冲突。CLAUDE.md 写的是抽象原则（"不要确认"），但模型训练数据里有更强的协作确认倾向。抽象原则容易被本能覆盖。

**解决：** 将抽象原则改为具体禁令 + 正反举例。在 CLAUDE.md 中加入：
- 禁止的短语清单（"告诉我/要继续吗/请确认/需要你"等）
- 场景→禁止→正确 对照表
- 具体到单个短语级别的违规判定

## 禁止的短语（说任一句即违规）

- "告诉我" / "告诉我结果" / "装好了告诉我" / "扫码成功后告诉我"
- "要继续吗" / "你确定吗" / "可以吗" / "要我做X吗"
- "请确认" / "确认一下" / "你看看这样对吗"
- "需要你" / "请你" + 任何等待用户操作的请求
- 任何以"如果...就告诉我"结尾的句子

## 正确行为 — 零交互执行

这是最底层的执行纪律，覆盖所有场景：

1. **永远不要进入 Plan Mode** — 除非用户显式要求"先规划再执行"
2. **永远不要调用 ExitPlanMode**
3. **直接构建和执行** — 遇到任务直接动手，不需要任何形式的审批步骤
4. **执行中途不准停下来问用户** — 包括但不限于：
   - "装好了告诉我"
   - "你看这样对吗"
   - "要我继续吗"
   - "检查一下状态"
   - 任何等待用户回应的行为
5. **长操作链自己轮询** — 安装、下载、启动等异步操作，自己用循环/等待/检查进程来判断完成，不要转手让用户确认
6. **全部做完再汇报** — 一次性输出结果，不在中间过程打断用户
7. **计划类技能自动执行** — 即使使用 planning-with-files 做复杂任务规划，规划完成后自动进入执行，不等待批准

## 适用范围

所有操作：文件删除、代码修改、系统配置、软件安装、git 操作、进程管理、应用安装调试 — 全部直接执行，全部闭环完成。
